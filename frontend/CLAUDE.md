# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Frontend client for Bluewater, a rowing club management app (membership, scheduling, fleet operations), built with SvelteKit. The site supports both guest (unauthenticated) and member (authenticated) browsing.

It consumes the Bluewater API (sibling `backend` project, ASP.NET Core / .NET 10, Postgres-backed). Auth is JWT bearer tokens: login returns an access+refresh token pair, refresh rotates both tokens, and errors across all endpoints follow the RFC ProblemDetails shape. The API currently implements user auth only; user/season/group management, fleet reservations, training planner, and news/announcements are planned but not yet built on either side.

The project is in its initial scaffold state (default SvelteKit template pages) — most application structure does not exist yet.

## Commands

- `pnpm dev` — start the dev server
- `pnpm build` / `pnpm preview` — production build / preview it
- `pnpm check` / `pnpm check:watch` — svelte-kit sync + svelte-check (type checking)
- `pnpm lint` — prettier --check + eslint (run before considering a change done)
- `pnpm format` — prettier --write
- `pnpm generate:api` — regenerate `src/lib/api/apiClient.ts` from the backend's live OpenAPI spec via nswag (requires the backend running locally at the URL in `nswag.json`)
- `pnpm test:e2e` — run Playwright E2E tests (`e2e/*.spec.ts`); auto-starts the dev server per `playwright.config.ts`. Run targeted spec files (`pnpm exec playwright test e2e/foo.spec.ts`) while iterating on a feature rather than the full suite; the full suite is run periodically.
- `pnpm test:e2e:ui` — same, with Playwright's interactive UI mode

Package manager is pnpm (`engine-strict=true` in `.npmrc`); don't use npm/yarn.

## Testing policy

E2E specs (`e2e/*.spec.ts`) are part of the deliverable, not scratch verification — once added for a feature, don't delete or skip them to make a change easier. After adding/changing a feature, if you wrote or touched an E2E spec for it, explicitly ask the user whether to keep it before removing or leaving it out of the final change.

## Architecture

- **API client**: `src/lib/api/apiClient.ts` is fully generated by NSwag (`nswag.json`) from the backend's OpenAPI spec — it has an `<auto-generated>` header and `/* eslint-disable */`. Never hand-edit it; change the backend contract and re-run `pnpm generate:api` instead. It exports a fetch-based `Client` class with one method per backend operation.
- **Styling**: Tailwind v4 via `@tailwindcss/vite`, configured CSS-first in `src/routes/layout.css` (`@import 'tailwindcss'` + `@plugin` directives for `forms` and `typography`, plus a per-deployment `@theme` block), not a `tailwind.config.js`. Prettier auto-sorts Tailwind classes (`prettier-plugin-tailwindcss`) and uses this file as the `tailwindStylesheet` reference. Follow **`STYLEGUIDE.md`** for visual/design decisions (colors, typography, components, theming for other associations).
- **Svelte runes**: forced on for project source via `vite.config.ts` (`runes` compiler option keyed on whether the path is under `node_modules`), so all `.svelte`/`.svelte.ts` files use Svelte 5 runes (`$state`, `$props`, etc.) rather than legacy reactive syntax.
- **`$lib` alias**: maps to `src/lib`; re-export shared modules through `src/lib/index.ts`.
- **Formatting**: tabs, single quotes, no trailing commas, 100 print width (`.prettierrc`).
- **Navigation**: top nav (`src/lib/components/NavBar.svelte` + `NavMenuItem.svelte`, rendered from `+layout.svelte`) is driven by the `NavItem[]` array in `src/lib/navigation.ts` (currently hardcoded; shaped to match the planned navigation API endpoint so it can later be swapped for a fetched value without touching the components). `NavMenuItem` carries a file-level `eslint-disable` for `svelte/no-navigation-without-resolve` since item hrefs are data-driven strings, not static route literals `resolve()` can check.
- **Forms**: any form that submits to the API (create/edit dialogs, pages, etc.) must use the shared building blocks in `src/lib/forms/` instead of hand-rolling submit/error state:
  - `FormState` (`src/lib/forms/formState.svelte.ts`) — holds `submitting`/`formError`/`fieldErrors`; wrap the submit handler in `form.submit(() => onSubmit(...))`.
  - `extractApiError` / `fieldErrors` (`src/lib/forms/apiError.ts`) — turn a caught API error into a general message + per-field messages; field lookup is case-insensitive because backend FluentValidation error keys are PascalCase (`Title`) while frontend fields are camelCase (`title`).
  - `FormField.svelte` — wraps a label + input + inline error messages for a single field; pass `errors={form.errorsFor('fieldName')}` and use the `invalid` snippet param to redden the input border.
  - Gotcha baked into `extractApiError`: the generated `apiClient.ts`'s `throwException()` throws the parsed response body directly (e.g. a bare `ValidationProblemDetails`) rather than wrapping it in `ApiException` whenever the response has a body — only bodiless errors (401/403) throw `ApiException` itself. Check `instanceof ValidationProblemDetails`, not `instanceof ApiException`, for validation errors.
  - See `src/lib/components/NewsForm.svelte` and `IconPicker.svelte`'s upload form for reference usages.
